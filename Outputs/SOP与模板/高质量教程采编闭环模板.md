# 高质量教程采编闭环模板

这份 SOP 用来判断一个教程、视频、案例或课程包是否值得进入知识库，并把它从候选材料加工成 Wiki 和正式 Output。

## 总流程

```text
候选池
  -> NotebookLM / 人工审片
  -> Clippings / Raw 入库
  -> 先找已有 Wiki 吸收
  -> 必要时新建 / 扩写 Wiki
  -> Output 生成或同步更新
  -> log.md 记录
```

## 1. 建候选池

文件位置：

```text
.tmp/<platform>-<topic>-candidates-YYYY-MM-DD.md
```

候选记录模板：

```markdown
## <标题>

- 来源：
- 作者 / 平台：
- 发布时间：
- 时长 / 篇幅：
- 主题：
- 初筛等级：must-review-in-notebooklm / watchlist / useful-evidence / skip
- 推荐理由：
- 可能补的 Wiki 缺口：
- 风险：
- 下一步：
```

## 2. 审片问题清单

放进 NotebookLM 后，优先问：

- 这个材料到底教人做出了什么具体东西？
- 有没有从 0 到可运行结果的完整步骤？
- 用了哪些工具、模型、平台和版本？
- 有没有报错、失败、调试和修复？
- 哪些步骤可以照做，哪些只是口头建议？
- 有没有成本、账号、API、部署、安全或合规要求？
- 有没有配套 prompt、课件、代码、截图或示例文件？
- 它和现有 Wiki 比，补了哪个缺口？
- 如果只能抽 3 个知识点进 Wiki，应该是哪 3 个？
- 值不值得入库？为什么？

用户反馈模板：

```markdown
材料：
结论：值得 / 不值得 / 只作旁证
最有价值的点：
最大问题：
适合进入哪个 Wiki 节点：
需要保留哪些原始文件：
下一轮希望多搜 / 少搜什么：
```

## 3. 入库规则

| 材料 | 放哪里 |
| --- | --- |
| 网页文章、blog、transcript、X、公众号剪藏 | `Clippings/` |
| NotebookLM 对话、用户总结、审片判断 | `Raw/NotebookLM/` |
| PDF、视频、截图、数据文件 | `Raw/` |
| 课程 prompt、课件、示例文件 | `Raw/CourseKits/<platform>--<topic>--<slug>/` |
| 临时搜索结果 | `.tmp/` |

不要创建旧目录：

- `Sources/`
- `Prompts/`
- `Reviews/`
- `Evals/`
- `Research/`
- `Taxonomy/`
- `Templates/`

## 4. NotebookLM 记录命名

```text
Raw/NotebookLM/<review-date>--<platform>--<topic>--<short-slug>.review.md
```

示例：

```text
Raw/NotebookLM/2026-05-28--youtube--vibe-coding--vibe-coding-full-course-gemini-3-1-antigravity-6hrs.review.md
```

## 5. CourseKit 命名

```text
Raw/CourseKits/<platform>--<topic>--<short-slug>/
  README.md
  prompts/
  assets/
```

`README.md` 至少包含：

- source URL。
- 对应 clipping。
- 对应 review note。
- 文件清单。
- 处理状态。
- 下一步。

## 6. 编译 / 回填 Wiki

编译顺序：

- 先读 NotebookLM / 用户审片记录。
- 再读 CourseKit 里的 prompt 和资产说明。
- 最后回查 Clippings 原文核对时间戳、步骤、原始表达。

写入 Wiki 前先做合并判断：

- 是否已有主页面能吸收这份材料。
- 是否只是补充旧页面的证据、案例、prompt 背后的任务逻辑、风险或检查项。
- 是否存在同一粒度下讲同一件事的旧页。
- 如果需要合并，按章节重排和融合，不把旧内容整篇 append 到尾部。
- 只有现有页面无法承载，且材料能独立支撑教程、SOP、案例、工具判断或 Prompt Output，才新建页面。

Wiki 页面检查：

- [ ] 解决什么实际问题。
- [ ] 适合谁，不适合谁。
- [ ] 输入、工具、步骤、输出。
- [ ] 可复用模板或检查清单。
- [ ] 来源证据。
- [ ] 常见坑、成本、限制和替代方案。
- [ ] 能生成哪些 Outputs。

## 7. 生成或同步 Output

Output 必须从 Wiki 生成，并在 frontmatter 或正文写明：

- [ ] 上游 Wiki 页面。
- [ ] 参考的 Raw / Clippings。
- [ ] 目标读者。
- [ ] 适用场景。
- [ ] 版本日期。

如果这轮 Wiki 维护合并、改名或改变了主结论，还要反向检查已有 Output：

- 是否还引用已删除或已合并的 Wiki 页面。
- 是否仍把旧页面当主入口。
- 是否需要补 `superseded`、维护备注或新版入口。
- 是否需要更新检查清单、课程讲义、案例拆解里的判断。

## 8. 更新状态

Clipping frontmatter 可更新：

```yaml
compile_status: partial
compiled_to:
  - Wiki/...
  - Outputs/...
remaining_value: medium
review_note: Raw/NotebookLM/...
```

注意：只更新 metadata，不改写 clipping 正文。

## 9. 更新 log.md

日志模板：

```markdown
- YYYY-MM-DD — 处理 `<材料名>`：基于 `<Clipping>`、`<Raw/NotebookLM>` 和 `<CourseKit>`，新增 / 更新 `<Wiki 列表>`，并生成 `<Outputs 列表>`。当前剩余疑点：`<疑点>`。
```

## 10. 最小验收

- [ ] 原始材料没有被改写、移动或覆盖。
- [ ] NotebookLM / 人工判断没有误放进 `Clippings/`。
- [ ] Prompt pack 没有绕过 Wiki 直接做成 Output。
- [ ] Wiki 不只是摘要，而是提炼成可复用方法。
- [ ] 新材料已先检查能否被已有 Wiki 主页面吸收。
- [ ] 合并后的内容经过选择、重排和融合，不是 append 到尾部。
- [ ] Output 写清上游 Wiki 和版本日期。
- [ ] 已有 Output 的旧引用和旧判断已反向检查。
- [ ] `log.md` 已记录本轮处理。
