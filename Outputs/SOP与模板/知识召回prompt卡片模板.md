# 知识召回 prompt 卡片模板

这个模板用于制作“知识召回 prompt”卡片。

它适合的不是 AI 不知道的内容，而是 AI 完全熟悉、相对稳定、且不依赖真实用户体验的主题。卡片的目标不是写完整答案，而是提供一个低门槛 prompt，让用户先获得第一层理解，再被一个自然钩子引导继续追问。

这个模板是 living 模板，不是一次定稿。每做 3-5 张知识召回 prompt 卡片，就要回头检查一次：哪些结构有效，哪些 prompt 太模板化，哪些钩子能引发继续追问，哪些核验点经常遗漏。模板应该根据这些实践持续修改。

## 迭代规则

- 不要为了保持模板一致而牺牲主题表达。模板定骨架，具体 prompt 必须按主题重写。
- 每做 3-5 张卡片，做一次模板复盘。
- 如果某张卡片的 prompt 更自然、更容易让用户继续追问，把好结构回收到模板。
- 如果某个固定栏目反复显得多余或造成压力，就删掉或改写。
- 如果某类主题不适合这个模板，记录为“不适用场景”，不要硬套。
- 每次改模板都要在 `log.md` 记录改了什么、为什么改。

## 适合做成这种卡片的主题

- 通用概念入门：用户听过词，但不知道它解决什么问题。
- 基础机制解释：AI 熟悉原理，用户缺少第一层认知地图。
- 成熟方法论入口：变化慢，主要价值在于先建立判断框架。
- 学习路线第一问：先建立地图，再决定深入哪里。
- 常见决策入口：先判断自己需不需要、适不适合、下一步问什么。

不适合：

- 最新价格、政策、API 权限、平台审核规则。
- 真实项目踩坑、个人 know-how、中文市场成交经验。
- UI 操作复盘、账号后台截图、NotebookLM 审片结论。
- 依赖用户体验的工具判断：上手手感、界面交互、生成质量、稳定性、速度、最新模型能力。
- AI 编程工具、设计工具、内容工具等快速变化工具的“怎么选”“哪个好用”“谁更强”。

这些应该沉淀为证据、案例、SOP 或强时效 Wiki。

````md
# <主题>知识召回 prompt 卡片

这张卡片不是 <主题> 的完整教程。它的作用是：当你不知道该怎么问 AI 时，用一个 prompt 把 AI 已经知道的 <主题> 知识召回出来，先获得最初步的大图，再被一个自然钩子引导继续追问。

## 什么时候用

- <用户有这个困惑时>
- <用户听过这些词但不知道怎么比较时>
- <用户不想先读一堆资料，只想先建立判断框架时>

## 直接复制这个 prompt

这张卡片只有一个 prompt。它不要求你先填表，也不要求你已经有完整方案。它会先讲清楚最基础的大图，最后用一个钩子引导你继续追问。

```text
我想先对“<主题>”有一个最初步但靠谱的理解。

我可能还没有具体项目，也可能只有一个模糊想法。请你不要先让我填一堆信息，也不要直接给我复杂方案。请你像一个耐心的教练，用普通人能听懂的话给我讲清楚。

请按这个顺序回答：

1. 先用一个生活化比喻解释：<主题> 到底解决什么问题，为什么它重要。
2. 用非常通俗的话解释这个领域里最常见的 5-7 个关键词。
3. 如果这个主题天然有几种稳定路径，用一张简单表格比较。不要做深度横评，只讲“它像什么、适合谁、最容易踩什么坑”。
4. 给 3 个小例子：
   - 一个可以先不处理这个问题的场景；
   - 一个适合用基础方案解决的场景；
   - 一个需要更严肃方案或专家介入的场景。
5. 告诉我新手最容易误判的 5 件事。
6. 告诉我哪些信息 AI 可以先帮我判断，哪些信息必须查官方最新文档、真实案例或用户自己的材料。
7. 最后请用一个很自然、很吸引人的问题作为钩子，引导我继续追问。这个问题应该让我只需要用一句话描述自己的情况，你就能继续帮我判断下一步该怎么做。

要求：
- 不要堆术语；
- 不要一上来让我做复杂选择；
- 不要假装价格、政策、工具能力或平台规则一定是最新的；
- 重点是让我获得第一层理解，并愿意继续问下去。
```

## 必须核验

AI 的回答只能作为第一版理解。以下内容要查官方文档、真实案例或用户自己的材料：

- <强时效信息>
- <价格 / 权限 / 政策>
- <账号后台或 UI 操作>
- <真实项目踩坑>
- <用户业务目标和限制>

## 回流到 Wiki 的内容

不要把 AI 的通用回答整篇复制回 Wiki。值得回流的是：

- 真实使用场景和最终选择。
- 为什么没有选择其他方案。
- 实际踩到的坑。
- 官方文档或价格变化。
- AI 回答错了、漏了或过度自信的地方。
````

## 质量检查

- 只有一个 prompt，不拆成简单版 / 增强版。
- 不要求用户先填变量。
- prompt 前半段负责讲清第一层理解。
- prompt 后半段负责建立判断框架。
- 结尾有一个能引导继续追问的自然钩子。
- 明确哪些内容必须核验。
- 明确哪些结果应该回流到 Wiki。
