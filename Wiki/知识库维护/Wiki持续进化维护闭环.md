# Wiki持续进化维护闭环

## 这个流程解决什么

你指出的问题是存在的：如果每次只把 `Raw/Clippings` 编译成新 Wiki 页面，而不回头整理旧页面，Wiki 会很快腐烂。

腐烂的表现：

- 同一个主题散落在多个页面，互相重复。
- 旧页面没有吸收新证据，结论变旧。
- 新材料只新增，不合并，导致入口越来越乱。
- Output 依赖旧 Wiki，但 Wiki 已经不代表最新判断。
- 索引像文件列表，不像知识地图。

所以 Wiki 不能只有“编译”动作，还必须有“维护、合并、重构、废弃、升降级”动作。

## 新工作流

把原来的单向流程：

```text
Raw / Clippings -> Wiki -> Outputs
```

升级成循环：

```text
Raw / Clippings
  -> 先找已有 Wiki 吸收
  -> 必要时才编译新增
  -> 维护审查
  -> 必要时用户讨论与决策
  -> 合并旧页
  -> 拓扑整理
  -> Output 更新
  -> 用户反馈
  -> Wiki 再维护
```

## 五类 Wiki 动作

| 动作 | 什么时候做 | 产出 |
| --- | --- | --- |
| 吸收 | 新材料能补强已有主题页、主干页、材料页或 Output 背后的 Wiki 主干 | 旧页新增证据、案例、反例、风险或结论更新 |
| 编译 | 新材料无法被现有主页面承载，且有独立输出价值 | 新 Wiki 页 |
| 合并 | 两页在同一粒度下讲同一件事，或新材料补强旧主题 | 合并后的主页面、旧页删除、reference 降级或备注 |
| 拓展 | 旧页结构正确，但需要加入新案例、工具、限制 | 更新章节、补证据、更新状态 |
| 修剪 | 页面过时、重复、太碎、没有输出价值 | 标记 deprecated、合并、删除入口 |

## 合并优先：大鱼吃小鱼模型

Wiki 不是每来一份材料就长一个新页面，而是像“大鱼吃小鱼”：已有主页面应该不断吸收新材料里的证据、案例、反例、检查项和判断。

默认顺序：

1. 原始材料进入后，先找能承载它的现有目录和主页面。
2. 如果材料只是补充旧主题，就把它吸收到旧页，不新建页。
3. 如果材料同时补强多个旧页，就拆成证据片段分别回填。
4. 如果材料提出了现有页面无法承载的新问题，并且能支撑教程、SOP、案例、工具判断或 prompt，才新建页。
5. 当一个主页面膨胀到结构明显拥挤、内部出现稳定子主题、且子主题能独立支撑 Output 时，再拆分。

新建页面的门槛：

- 它解决的问题和现有主页面不同。
- 它的读者、使用场景、证据类型或 Output 形态明显不同。
- 它已经有足够材料支撑，不是两三行定义。
- 它拆出去后能让主页面更清晰，而不是制造新的入口噪音。

拆分页面的门槛：

- 主页面已经过长，读者难以扫描。
- 某个子主题反复被新材料补强，已经形成稳定结构。
- 子主题有自己的案例、SOP、prompt、工具判断或能支撑独立课程 Output。
- 拆分后，原主页面仍保留路线图、结论摘要和到子页的清晰入口。

## 同粒度合并原则

同一粒度下讲同一件事的页面应该被合并。判断时先看页面承担的对象和输出，而不是文件所在目录。

应该合并：

- 两个页面都围绕同一个 prompt、同一门课的同一章节、同一个案例或同一个工具判断。
- 一个页面叫结构分析，另一个页面叫资产页，但读者实际要解决的是同一个问题。
- 两页都能生成同一类 Output，且没有明显的主从关系。

暂不合并：

- 一个是主干页，一个是来源材料页，粒度不同。
- 一个是检查清单，一个是可执行 prompt，使用方式不同。
- 一个是通用框架，一个是真实案例，应该互相链接而不是合并。

合并后只保留一个主页面；旧页的有价值段落并入主页面，所有入口和 Outputs 改指向主页面，`log.md` 记录合并原因。

## 合并执行标准

合并不是把旧页面整篇 append 到主页面尾部。合并必须做选择、重排和融合。

执行步骤：

1. 先判断主页面承担的核心任务和读者路径。
2. 把被合并页拆成判断、步骤、案例、prompt、检查清单、风险、证据锚点和缺口。
3. 只保留能增强主页面的内容；重复定义、过时表述、低价值铺垫直接丢弃。
4. 把保留内容放进主页面最合适的章节，而不是集中堆到末尾。
5. 如果被合并页改变了主页面结论，要重写相关结论，而不是只新增一个补充段落。
6. 合并后检查主页面结构是否更清楚；如果更长但更乱，说明合并失败，需要再重排。
7. 最后更新入口、反向链接、Outputs 上游、Raw/Clippings metadata 和 `log.md`。

合并后的主页面应该变得更强：结论更清楚，证据更完整，后续新材料更知道该往哪里放。

## AI 已知知识的处理方式

Wiki 不是要替代 AI。对于 AI 本来完全熟悉、相对稳定、且不依赖真实用户体验的主题，维护目标不是复写一篇完整百科，也不是直接把 Wiki 变成 prompt 页，而是整理出这个主题的理解路径、问题框架、来源里值得学习的表达和本库需要补充的证据。需要交付给用户使用时，再从 Wiki 生成知识召回 prompt Output。

判断规则：

| 主题类型 | Wiki 处理方式 | 下游 Output |
| --- | --- | --- |
| AI 完全熟悉、稳定、用户不知道怎么问 | 问题框架、讲解路径、关键概念、追问顺序 | 知识召回 prompt 卡片 |
| AI 能答但容易泛泛 | 本库判断、真实输入输出、检查清单、边界条件 | SOP、模板、发号施令 prompt |
| AI 不擅长、依赖体验或容易过时 | 证据 / 案例 / 判断页 | 真实踩坑复盘、工具台账、强时效检查清单 |
| 混合型主题 | 通用框架 + 真实案例 + 本库判断 | 知识召回 prompt、教程、案例拆解 |

知识召回 prompt Output 的最低结构：

- 使用场景：用户什么时候复制这个 prompt。
- 主 prompt：一段能先通俗解释大图、再给关键判断框架、最后用钩子引导追问的提示词。
- 核验点：哪些回答必须查官方文档、最新价格、真实案例或用户自己的材料。
- 上游 Wiki：这个 prompt 来自哪些 Wiki 页面。
- 回流规则：如果用户用这个 prompt 得到有价值结论或踩坑，应回写到 Wiki 的证据 / 案例 / 判断区。

## AI 自主维护与用户讨论边界

Wiki 是主知识资产，但维护它不能变成把大量琐碎判断交给用户。默认原则是：AI 能判断的事情，AI 自己完成；只有涉及用户目标、私有材料、业务取舍或重大方向时，才进入讨论。

AI 应该自主完成：

- 这个新页面是否和已有页面重复？
- 它应该成为独立页，还是并入旧页？
- 是否存在同一粒度下讲同一件事的页面；如果存在，合并成一个主页面。
- 它改变了旧页面的结论吗？
- 它能补哪条路线图节点？
- 它有没有让某个旧 Output 需要更新？
- 是否需要新建索引、改名或移动？
- 是否有旧页面应该降级为 reference？

这些是 AI 的内部检查清单，不应默认逐条询问用户。检查后能直接处理的，就直接维护并记录到 `log.md`。

需要用户参与讨论的情况：

| 类型 | 什么时候问用户 | AI 先做什么 |
| --- | --- | --- |
| 私有材料 | 需要用户账号、付费资料、NotebookLM 审片记录、课程文件或真实业务数据 | 说明缺什么、为什么缺、拿到后能编译成什么 |
| 业务取舍 | 需要确定面向新手还是进阶、国内还是海外、教程还是商业化、课程还是 SOP | 给出 2-3 个方向和利弊，不问琐碎实现细节 |
| 重大结构 | 目录要大范围重构，或一个专题要从材料库转成课程体系 / 工具库 / 案例库 | 先提出结构草案和迁移影响 |
| 证据冲突 | AI 查证后仍无法判断哪个来源更可靠或更符合用户目标 | 列出冲突证据和推荐判断 |
| 投入优先级 | 下一阶段应该重点吸收哪类材料，且不同方向都会消耗用户时间 | 给出优先级建议，而不是开放式追问 |

执行原则：

- 小修小补、补索引、补链接、状态标记、常规合并和常规改名直接执行。
- 中等规模维护先执行可逆的小步改动，并在 `log.md` 说明。
- 大范围重构、结论反转、正式 Output 返工、专题方向切换，先给用户决策清单。
- 如果只是缺公开资料，AI 先自己搜索或从已有 Raw / Clippings 查证；只有缺用户私有材料或人工判断时才请求用户补充。

## 页面状态建议

| 状态 | 含义 |
| --- | --- |
| `seed` | 初始框架，证据少，暂不生成正式 Output |
| `compiled` | 已经能支撑教程、SOP、案例或工具判断 |
| `living` | 主干页，会持续吸收新材料和更新结论 |
| `needs_merge` | 与其他页面重复，需要合并 |
| `needs_refresh` | 依赖时效信息，需要重新核验 |
| `needs_decision` | 目录走向、定位或结论取舍需要用户决策 |
| `deprecated` | 结论过时或被新页面取代 |

## 主干页和材料页分工

为了避免页面爆炸，需要区分：

- **主干页**：长期维护的知识入口，例如 `Vibe Coding专题路线图`、`Vibe Coding安全审计清单`。
- **材料页**：来自单个课程/案例的编译结果，例如某个 SaaS 案例。
- **证据页**：只做旁证，不承担主结论。
- **Prompt Output**：从 Wiki 生成的交付产物，包括发号施令 prompt 和知识召回 prompt；它不承担 Wiki 主干结构。

主干页应该持续吸收材料页结论；材料页保留来源细节；Prompt Output 负责让用户复用和追问，不负责沉淀主结论。每次 Wiki 主干变化后，要检查相关 prompt Output 是否需要同步更新。

## 建议节奏

### 每次编译后：小整理

- 更新相关索引。
- 更新路线图节点。
- 给旧页面补链接。
- 判断是否重复。
- 写出资料缺口和可能的用户决策点。
- 记录 `log.md`。

### 每 5-10 篇新 Wiki：主题整理

- 搜索同主题页面。
- 合并重复段落。
- 更新主干页。
- 标记 `needs_merge` 或 `deprecated`。
- 检查 Outputs 是否需要再生成。
- 输出一次专题维护报告：已自主处理什么、建议下一阶段吸收什么、哪些少数高影响决策才需要用户确认。

### 每月或每个专题阶段：拓扑整理

- 重新看 `Wiki/README.md` 和专题 README。
- 把碎片页面聚成主干。
- 清理没有输出价值的入口。
- 更新专题路线图。
- AI 先给专题走向建议；只有当方向会切换为课程、案例库、工具库或正式产品化输出时，再请用户确认。
- 写一条 log 记录结构变化。

## 维护 Prompt

```text
请对这个 Wiki 专题做一次维护审查。

范围：
{WIKI_SCOPE}

请检查：
1. 哪些页面重复或应该合并；
2. 哪些页面已经过时或需要重新核验；
3. 哪些新页面改变了旧主干页结论；
4. 哪些页面应该升格为主干页；
5. 哪些页面应该降级为 reference 或 deprecated；
6. 哪些 Outputs 需要根据 Wiki 更新；
7. 索引和路线图是否仍然清晰。
8. 当前缺哪些关键资料或证据；公共资料由 AI 自行查证，私有/付费/人工审片材料才请求用户提供；
9. 下一阶段最值得吸收哪些知识、话题、案例或工具，并给出 AI 推荐优先级；
10. 哪些高影响目录走向、主干页定位或输出形态超出 AI 自主决策范围，才需要询问用户。

默认先执行低风险维护；涉及大范围重构、结论反转、正式 Output 返工或专题方向切换时，先输出决策清单。
每条建议都要说明原因、涉及页面和风险。
把建议分成两类：
- AI 已处理或可直接处理的维护；
- 少数需要用户确认的高影响结构或方向决策。
```

## 这次对当前 Vault 的改进建议

1. `Prompt` 资产应该作为 `Outputs/` 交付产物维护，不恢复旧顶层 `Prompts/`，也不在 Wiki 下新增 prompt 目录。
2. `Vibe Coding专题路线图` 应该成为 living page，持续吸收支付、平台 API、中文获客和 prompt 背后的任务逻辑。
3. 每次完成某门课程编译后，不应只生成 Output，还要做一次“旧页影响检查”。
4. 所有强时效页面，例如支付和平台 API，应加 `verified` 日期和 `needs_refresh` 机制。
5. `log.md` 不能只记录新增，也要记录合并、改名、废弃和结论变化。
6. Vibe Coding 专题下一阶段由 AI 先给优先级建议；只有涉及课程化、商业化定位或目录大重构时，再和用户讨论。
