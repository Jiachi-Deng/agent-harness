# Vibe Coding商业化与定价包装

## 这个页面解决什么

课程最后一段把问题从“怎么做软件”转到“怎么把 AI 做出来的软件卖出去”。它的核心判断是：当软件生产成本下降后，单纯会写代码不再稀缺，稀缺的是分发、客户理解、包装、交付、维护和定价。

这里的内容有强讲师个人经验色彩，尤其是营收数字和海外报价，只能作为商业思路来源，不能直接当作本地市场定价标准。

## 产品包装

AI 写出来的项目不能直接交付给客户。需要把它包装成客户能理解、能使用、敢付钱的商品。

标准包装至少包含：

| 包装层  | 内容                            |
| ---- | ----------------------------- |
| 在线演示 | Netlify/Vercel 演示站，客户能直接点开体验  |
| 功能清单 | 基础功能、高级功能、增值功能分层展示            |
| 使用文档 | 普通用户怎么注册、登录、操作、导出             |
| 部署文档 | 环境变量、Supabase、Netlify、域名、更新流程 |
| 安全报告 | 密钥、RLS、服务端验证、依赖、认证中间件检查结果     |
| 版本记录 | 当前版本、更新日期、已知限制、回滚方式           |

## 三种变现模式

| 模式 | 适合产品 | 收入特点 | 风险 |
| --- | --- | --- | --- |
| 定制开发 | 企业网站、客户仪表盘、内部工具、垂直 SaaS 原型 | 单笔客单价高，适合拿第一批案例 | 容易陷入无限改需求和免费售后 |
| 订阅制 SaaS | 线索挖掘、缩略图生成、内容分发、营销自动化 | 有复利，能形成长期收入 | 需要获客、运维、客服、安全和合规 |
| 模板售卖 | 作品集、Landing Page、简单后台、行业模板 | 低价走量，一次开发多次售卖 | 竞争激烈，容易同质化 |

## 从卖小时到卖项目

`The AI Offer You Can Sell Tomorrow Morning` 补充了一条更适合新手的商业化阶梯：不要一开始就卖 5,000 美元项目或月费 retainer，先卖一小时咨询 / 搭建 / 教学。

这条阶梯可以理解为：

| 阶段 | 卖什么 | 作用 |
| --- | --- | --- |
| Rung zero | 一小时 AI Operating System 设置、AI 工具教学、业务流程梳理 | 降低信任门槛，训练表达，进入客户真实业务 |
| Audit | 付费诊断、workflow map、自动化机会清单 | 把模糊需求变成项目范围 |
| Project | 聚焦一个 workflow 或内部工具 | 用明确 ROI 证明能力 |
| Retainer | 持续维护、优化、自动化和培训 | 适合已有信任和稳定需求的客户 |

对 Vibe Coding 商业化的启发是：早期不要把“我能用 AI 做 app”包装成大而全的承诺。更安全的入口是“我先陪你把一个业务场景跑通，让你理解 AI 能怎么进入你的工作”。

这个小时不是单纯按时间卖劳动力，而是同时完成四件事：

- 建信任：客户看到你能解释复杂工具，能管理他的焦虑。
- 做发现：你看到他的数据源、流程、团队抱怨和重复动作。
- 做 scoping：下一步项目范围自然浮现。
- 做筛选：不愿意投入时间、没有真实痛点或不愿配合的人，不适合升级成项目客户。

## 定价原则

课程强调的核心不是按工时报价，而是按价值报价：

- 客户因为这个工具省了多少钱。
- 客户因为这个工具多赚多少钱。
- 客户因为这个工具节省了多少时间。
- 你收取其中一小部分价值，而不是按你花了几小时来算。

对 Vibe Coding 尤其重要：AI 让开发时间缩短，如果还按小时报价，越高效越吃亏。

## 交付和售后边界

商业化最容易翻车的不是做不出，而是交付边界不清：

- Bug 修复和新增功能要分开。Bug 可以在维护期内免费，新功能要重新报价。
- 所有需求必须转成书面功能清单，避免“顺便改一下”无限扩大。
- 每次改动都要本地测试、线上测试、安全复检。
- 客户要拿到使用文档，而不是只拿一包代码。

## 需要本地化校准

- 海外美元价格不能直接搬到中文市场。
- 国内平台接口、自动化、爬虫和群发邮件/私信有更复杂的合规边界。
- 面向小企业时，“能不能有人帮我部署和维护”通常比“源码多高级”更重要。
- 客户愿意付费的不是 AI 技术，而是明确的业务结果：获客、节省时间、减少人力、提高转化。
- 一小时咨询、audit、project、retainer 的价格阶梯需要本地化；当前海外报价只能作为包装思路，不应直接套用。
