# 后端与数据库选择

## 页面定位

这页不是 Firebase / Supabase / Neon / Convex 的完整百科，也不是强时效工具横评。后端、数据库、认证、存储这些通用概念，AI 本来就能解释得不错；Wiki 更应该保留的是：新手该按什么顺序理解这个问题、材料里哪些表达和案例值得复用、哪些结论必须靠真实项目或官方资料核验。

这页负责沉淀“怎么判断要不要后端、后端到底包含哪些能力、什么时候必须补证据”。可复制的知识召回 prompt 已作为 Output 维护：

- [[Outputs/SOP与模板/后端选择知识召回prompt卡片|后端选择知识召回 prompt 卡片]]

## 这个问题为什么重要

Vibe Coding 新手很容易把“做 Web App”直接等同于“先选数据库”。这会导致两种常见错误：

- 过早接后端：一个静态展示页、计算器、小工具、本地原型，还没验证价值就开始折腾登录、权限和部署。
- 过晚接后端：已经涉及用户数据、上传、支付、多人协作或 API secret，却还把数据和 key 放在前端。

更好的顺序是：先看 app 需要保存什么、谁能访问、是否有账号、是否有文件、是否需要服务端保护密钥，再决定后端方案。

## 判断框架

### 1. 先判断要不要后端

可以先不用后端的情况：

- 纯展示型 Landing Page。
- 个人使用的小工具，数据不需要跨设备保存。
- 只验证交互和 UI 的原型。
- 可以先用本地存储或手工表格过渡的早期想法。

通常需要后端的情况：

- 用户需要注册、登录、找回账号。
- 不同用户看到不同数据。
- 需要保存表单、订单、评论、文件或生成结果。
- 需要隐藏 OpenAI / Stripe / Apify 等服务端密钥。
- 需要积分、订阅、支付、审核、队列或多人协作。

### 2. 后端不是一个东西

新手可以先把后端拆成几块能力，而不是一上来比较品牌：

| 能力 | 解决什么问题 | 新手最常见误判 |
| --- | --- | --- |
| Authentication | 谁是用户，用户如何登录 | 以为登录做好了就等于数据安全 |
| Database | 数据保存在哪里，如何查询 | 以为有数据库就自动有权限控制 |
| Storage | 图片、PDF、视频等文件放哪里 | 忘记文件访问权限和公开链接风险 |
| Server / API routes | 哪些逻辑必须在服务端执行 | 把 secret key 放进前端代码 |
| Realtime | 多人协作、聊天、状态同步 | 过早为了“高级感”增加复杂度 |
| Payment / credits | 收费、积分、用量和账单 | 没有处理 webhook、重试和滥用成本 |

### 3. 先按 app 行为选，不按工具名选

粗略理解即可，不要把下表当最新横评：

| 工具 | 更像什么 | 适合的第一层场景 | 容易踩的坑 |
| --- | --- | --- | --- |
| Firebase | 一套完整 app 后端工具箱 | 新手快速做登录、数据库、文件上传、部署演示 | rules、authorized domains、价格和厂商绑定 |
| Supabase | Postgres + Auth + Storage 的成套后端 | 想用 SQL/Postgres，并希望有开源生态 | RLS、schema、client/server key 边界 |
| Neon | 托管 Postgres 数据库 | 已有后端/API，只缺数据库 | 它不是完整 app builder，auth/storage 还要另配 |
| Convex | 数据、函数和实时体验紧耦合的平台 | 实时协作、快速迭代、全栈 TS 心智 | 平台心智和迁移边界需要理解 |
| Airtable / Notion / Google Sheets | 轻量数据后台或人工运营表 | MVP、内部工具、低频表单和内容管理 | 权限、规模、自动化稳定性和 API 限制 |

## 从当前材料得到的证据

[[Clippings/Vibe Coding for Beginners (Full Course 2026)|Vibe Coding for Beginners]] 这门课使用 Firebase authentication、Firestore、Storage，并涉及 Vercel authorized domains 和 console 调试。它说明 Firebase 这类一体化后端适合教学和新手从本地 demo 走到可部署 app，因为登录、数据、文件和部署校验能在同一条路径里串起来。

但这个案例不能推出“所有新手都应该选 Firebase”。它只能作为一个教学证据：当课程目标是从 0 快速完成一个有账号、有数据、有上传、有部署的 Web App，一体化 BaaS 能降低初学者的拼装成本。

## 需要重点沉淀的增量

后续不要在这页补通用百科，重点吸收这些材料：

- 真实项目为什么选 Firebase / Supabase / Neon / Convex 或为什么放弃。
- AI 生成 rules / RLS / schema / API routes 的错误案例。
- Auth、storage policy、server key、callback domain 的真实配置坑。
- 官方价格、免费额度、地区、API 和部署限制变化。
- 从纯前端 demo 升级到带登录、数据库、文件上传或支付时的 UI 操作复盘。
- 经过验证的数据模型、权限模型和上线检查清单。

## 核验点

AI 可以给第一版方向，但这些必须人工或联网核验：

- Firebase / Supabase / Neon / Convex 的最新免费额度、计费、地区和产品限制。
- Auth、RLS、security rules、storage policy 的官方写法和默认行为。
- 前端可暴露 key 与服务端 secret 的边界。
- Vercel / Netlify / Cloudflare 的环境变量、callback domain、serverless runtime 限制。
- 付费 AI API、爬虫、邮件、文件存储的滥用成本和限流策略。
- 真实用户数据、客户数据、支付数据是否涉及合规要求。
